# 实验报告：任务 4 - 完成用户注册与登录功能

## 1. 实验目的
本任务旨在实现用户的注册与登录业务逻辑，引入 DTO（数据传输对象）以及 Service（业务逻辑层）以保持架构的纯洁性，并使用 MyBatis-Plus 框架直接与真实的 PostgreSQL 数据库打通，实现数据的落地与持久化。同时，使用 Postman 进行完整业务流联调测试。

## 2. 核心步骤与代码实现

### 2.1 准备实体类 (Entity) 与数据传输对象 (DTO)
创建与数据库 `sys_user` 表结构严格一一对应的实体类 `User`，以及专门用于接收前端 JSON 数据的传输对象 `UserDTO`，避免将底层实体直接暴露给表现层。
- **Entity**: `com.stu.helloserver.entity.User`
- **DTO**: `com.stu.helloserver.dto.UserDTO`

### 2.2 完善全局状态码 (ResultCode)
在 `ResultCode` 枚举类中追加注册和登录相关的业务状态码：
```java
USER_HAS_EXISTED(4001, "该用户名已被注册"),
USER_NOT_EXIST(4002, "该用户不存在"),
PASSWORD_ERROR(4003, "账号或密码错误");
```

### 2.3 配置 MyBatis 与 Mapper 层
在 `application.yml` 中配置 PostgreSQL 数据库连接。同时创建 `UserMapper` 接口继承 `BaseMapper<User>`，获得 MyBatis-Plus 提供的强大单表增删改查能力。

### 2.4 编写业务层 (Service) 与控制层 (Controller)
编写 `UserService` 接口及其实现类 `UserServiceImpl`。在 `UserServiceImpl` 中，我们通过注入 `UserMapper`，实现了与真实数据库的交互，处理了用户是否存在校验、密码比对、以及新用户落库等核心逻辑。

在 `UserController` 中，通过依赖注入引入 `UserService`，接通注册 (`POST /api/users`) 和登录 (`POST /api/users/login`) 接口，并提供获取用户信息接口用于测试拦截器。

### 2.5 修复全局异常处理与拦截器配置
- 修复了 `GlobalExceptionHandler` 中由于 `Result.error` 方法缺少重载导致的编译报错。
- 完善了 `AuthInterceptor`，确保注册与登录请求能够正常放行，同时对其他受保护的接口执行严格的 Token 校验。
